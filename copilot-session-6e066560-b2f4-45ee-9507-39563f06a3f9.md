**Never committed to the repo.** `docs/index.html` is built *inside* the GitHub Actions runner and uploaded as an artifact directly. It's ephemeral — not in git history. The site updates every time `pages.yml` runs (every push to `main`).

## How It's Supposed to Work

**Phase 1 — CD Practice (workflow setup):**
1. Student branches from `main`
2. Copies `_template-publish.yml.example` → `.github/workflows/\<name\>-publish.yml`
3. Sets `STUDENT_BRANCH` and `OUTPUT_FILENAME` (e.g. `tsteele/rocket-page`, `tsteele.html`)
4. PRs that workflow file into `main` ✅

**Phase 2 — Rocket editing:**
1. Student works on their branch (e.g. `tsteele/rocket-page`) and edits `git-rocket.js`
2. Every push to that branch triggers **their** publish workflow
3. The workflow builds `docs/tsteele.html` and uses `peter-evans/create-pull-request` to open/update a PR that commits that file to `main`'s `docs/` folder
4. When that PR is merged → `pages.yml` redeploys GH Pages → student visits `https://\<org\>.github.io/git-rocket/tsteele.html` 🚀

---

## Current Issues

| Issue | Impact |
|---|---|
| `peter-evans/create-pull-request@v7` commits `docs/\<name\>.html` to `main` | ✅ This is the intent — it works |
| `pages.yml` doesn't commit `docs/` either — it builds ephemerally | ⚠️ **Conflict:** `pages.yml` runs `build:docs` which regenerates `docs/index.html` only — it will NOT pick up student `.html` files unless they're already committed to `main`'s `docs/` folder |
| GH Pages must be enabled manually | One-time repo-owner setup required |

## The Key Insight ✅

The student `.html` files **are committed to `main`** (via the automated PR). So when `pages.yml` runs, `./docs` contains both the regenerated `index.html` **and** all student files. The upload-artifact step picks up the whole `docs/` folder. **It should work** — as long as `pages.yml` doesn't clobber the folder (it doesn't; it just adds `index.html`).

**Summary: The design is sound.** The only real prerequisite is that GH Pages source must be set to "GitHub Actions" in repo settings.

### 💭 Reasoning

*The student is asking whether the publish workflow needs to create a PR into main, or if it can just publish directly to GH Pages like pages.yml does.

The issue is: pages.yml is the only workflow that deploys to GH Pages (it has the `pages: write` permission and uses `actions/deploy-pages`). Student publish workflows run on their own branches, not main.

GH Pages deployment via Actions requires the `deploy-pages` action, which can only run once at a time (concurrency group). Multiple workflows trying to deploy would conflict.

The question is: can the student's workflow, running on their branch, deploy directly to GH Pages without going through main?


The real constraint is that GH Pages deployments replace the entire site rather than adding to it—each deployment wipes out what came before. The PR-to-main approach works because it accumulates all student files in `docs/` before pages.yml deploys them together, but if students deployed directly from their branches, they'd each overwrite everyone else's content.*

<sub>⏱️ 7m 3s</sub>

### 💬 Copilot

**It can't — not cleanly.** Here's why:

`actions/deploy-pages` **replaces the entire GH Pages site** with whatever artifact you upload. It's not additive. So if student A deploys directly, they wipe out student B's page.

The PR-to-`main` approach works *because* `docs/` in `main` accumulates all student files over time. Then `pages.yml` uploads the whole folder as one artifact → one deployment → everyone's page survives.

```
Student branch push
  → build docs/tsteele.html
  → PR commits it to main/docs/
    → pages.yml triggers
      → uploads ALL of docs/ (index.html + tsteele.html + sbennett.html + ...)
        → deploys everything at once ✅
```

If you tried direct deploy per student:
```
Student A deploys → site = { tsteele.html }
Student B deploys → site = { sbennett.html }  ← tsteele.html gone ❌
```

**The PR into main is load-bearing.** It's also pedagogically intentional — students experience the full CD pipeline: branch → build → automated PR → review/merge → live site.
